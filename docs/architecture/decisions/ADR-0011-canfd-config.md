# ADR-0011: CANFD channel configuration for the Node A application

**Status:** accepted · **Date:** 2026-06-28

## Context

M2 brings up CAN on the Node A application (seam 2 of the M2-1 ordering). The TRAVEO™ T2G
CAN peripheral is a **Bosch M_CAN-derived CANFD IP**: it has a **dedicated message RAM that
must be partitioned** before the channel works, separate **nominal** (arbitration) and
**data** bit timings, and a rich interrupt-source set. This ADR fixes the channel config and
the host/target split; the message *content* and ID allocation are owned by `shared/messages`
and ADR-0002, and the interrupt-priority rules by ADR-0010 D3.

Scope for M2 is minimal-but-real: **one channel**, RX→ISR→queue→task, TX, with a loopback
**echo** demo (there is no Node B yet — real sensor-report→command routing arrives in M5).

## Decisions

### D1. One channel, **full CAN FD** with bit-rate switching

| Phase | Bitrate | Sample point | Note |
|---|---|---|---|
| Nominal (arbitration) | **500 kbit/s** | ~80% | conventional body-bus rate |
| Data (BRS) | **2 Mbit/s** | ~75% | flexible-data-rate payload phase |

- **FD frames, BRS on.** We bring up the full CAN FD timing now (both bit timings), not a
  classic-only channel — the data-phase timing is designed and proven on silicon in M2 rather
  than deferred.
- Prescaler / `seg1` / `seg2` / `SJW` for **both** phases derive from the CANFD peripheral
  clock and are produced by the ModusToolbox device configurator (`design.modus` →
  `cycfg_peripherals`). The two sample points above are the design targets to confirm against
  the generated timing.
- **Consistency with the deferred FBL knock window (ADR-0008 D2):** the channel, bitrates,
  and the fixed knock/diagnostic ID are single-sourced (with the IDs in `shared/messages` /
  ADR-0002) so the FBL knock path, when implemented, uses identical parameters. M2 sets these
  shared parameters; it does not pick numbers the FBL would have to contradict.

### D2. Message-RAM partition (minimal, single channel)

The M_CAN message RAM is a scarce, explicitly-partitioned resource — the CAN analogue of the
flash / `.noinit` "the silicon has structure you must plan for" lesson.

| Section | M2 allocation |
|---|---|
| Standard ID filters (SIDFC) | 1–2 elements (accept the gateway RX IDs; accept-all for the echo demo) |
| RX FIFO 0 (RXF0C) | ~8 elements — the RX path |
| TX buffers (TXBC) | 1 dedicated TX buffer |
| RX buffers / RX FIFO 1 / TX event FIFO | **0** — unused in M2 |
| Element size | **64-byte data** (FD-capable elements) |

**Footprint must fit the channel's message-RAM allotment.** With 64-byte FD elements a
RX FIFO 0 element is ~72 B (8 B header + 64 B data), so ~8 RX elements ≈ 576 B, plus 1 TX
buffer ≈ 72 B and 1–2 filter elements — on the order of ~700 B for this channel. The M_CAN
message RAM is small and shared across channels, so this is a **sizing constraint to confirm
against the part's CANFD message-RAM size before bring-up**, not a free allocation. If it does
not fit, the first lever is RX FIFO depth (8 → 4).

The partition (section start offsets, element counts, element size) is generated by the
configurator and wrapped behind the HAL (D5), keeping generated vendor code out of
host-testable logic.

### D3. RX path — RX FIFO 0 + the `RF0N` interrupt

- Accepted frames land in **RX FIFO 0**; `Cy_CANFD_Init` enables the **`RF0N` (RX FIFO 0
  New-message)** interrupt on **interrupt line 0** and runs it at **level 5 (`0xA0`)** —
  kernel-aware per ADR-0010 D3 (numerically ≥ `configMAX_SYSCALL_INTERRUPT_PRIORITY` = `0x40`),
  set **explicitly** (never the reset-default level 0).
- **TRAVEO NVIC mux (silicon-verified — the seam-2 bring-up trap).** On TRAVEO T2G the CM4
  reaches peripheral interrupts through an **8-channel NVIC mux** (`NvicMux0..7`); the CANFD
  IRQ enum (`canfd_0_interrupts0_1_IRQn` = 58) is a **system-interrupt index, not a CM4 NVIC
  line**. `cy_stc_sysint_t.intrSrc` must **pack a mux channel** (`<< CY_SYSINT_INTRSRC_MUXIRQ_SHIFT`)
  with that system interrupt, and `NVIC_EnableIRQ` takes the **mux channel** (we use `NvicMux3`).
  Passing the bare IRQn silently maps it to `NvicMux0` and enables a non-existent line → the
  FIFO fills and RF0N sets in the peripheral but **no interrupt reaches the CM4** (the classic
  "init OK, no RX"). This is distinct from PSoC6 CM4, which has direct NVIC lines.
- The PDL `Cy_CANFD_IrqHandler` (called from the channel ISR) drains FIFO 0 and invokes the
  config's `rxCallback` per frame; our callback packs a `can_raw_frame_t` and pushes it to
  `raw_frame_q` with `xQueueSendFromISR`. The ISR then calls `portYIELD_FROM_ISR(woken)` so
  `CAN_CyclicTask` runs on ISR exit (ADR-0010 D5).
- `CAN_CyclicTask` dispatches on `id` to the existing `unpack_*` (host-testable
  `shared/messages`), producing a `body_msg_t` for `App_CyclicTask` / `bodyctl_step`. For the
  M2 echo demo it re-transmits the received frame to prove the loop.

```mermaid
flowchart LR
  BUS[CAN bus] -->|FD frame| FIFO0[RX FIFO 0]
  FIFO0 -->|RF0N IRQ @ lvl 5| ISR[CAN RX ISR<br/>drain + xQueueSendFromISR]
  ISR -->|raw_frame_q| TASK[CAN_CyclicTask<br/>dispatch + unpack_*]
  TASK -->|body_msg_t| APP[App_CyclicTask<br/>bodyctl_step]
  TASK -. M2 echo .->|TX buffer| BUS
```

### D4. TX path

A single dedicated **TX buffer**: a task builds a `can_raw_frame_t`, the HAL `send()` writes
it into the TX buffer and requests transmission. No TX event FIFO in M2 (no completion
bookkeeping needed for the echo).

### D5. HAL seam + host fake (ADR-0001)

A thin interface keeps the routing free of vendor headers:

```c
/* shared/can/include/can_hal.h — plain types, no vendor headers */
typedef struct {
    uint32_t id;        /* 11- or 29-bit, per flags */
    uint8_t  flags;     /* IDE / RTR / FDF / BRS bits */
    uint8_t  len;       /* DLC-decoded byte count, 0..64 */
    uint8_t  data[64];
} can_raw_frame_t;

typedef struct {
    int (*init)(const can_cfg_t *cfg);
    int (*send)(const can_raw_frame_t *f);
} can_hal_if_t;
```

- **Target impl** wraps the ModusToolbox CANFD PDL; the ISR is target-only and owns
  `raw_frame_q`.
- **Host fake**: `send()` records frames; `fake_can_inject_rx()` pushes `can_raw_frame_t`s as
  if received. The **whole RX-routing path tests on host** — inject a raw frame, assert the
  right `unpack_*` ran and the right `bodyctl` output / echo resulted — no hardware, no
  FreeRTOS (the task adapter is the only target-only glue).

## Host/target split (ADR-0001)

| Host-testable | Target-only |
|---|---|
| `id` → `unpack_*` routing over `can_raw_frame_t` | CANFD PDL init + message-RAM partition |
| `bodyctl` reaction to decoded messages (ADR-0010 D6) | `RF0N` ISR, FIFO drain, TX request |
| Echo policy (which frames re-TX) | NVIC priority set; bit-timing register values |

## Consequences

- (+) Full FD (incl. BRS) is proven on silicon in M2, not deferred.
- (+) Message-RAM partitioning is explicit and configurator-generated; the HAL hides it.
- (+) The RX path reuses the existing `shared/messages` decode and stays host-testable end to
  end.
- (−) Two bit timings to get right (nominal + data) instead of one — the cost of FD now.
- (−) Bit-timing register values and message-RAM offsets depend on the configurator output /
  CANFD clock, confirmed on board.

## Alternatives considered

- **Classic CAN 2.0 (or FD without BRS) for M2:** one bit timing, simpler bring-up — but
  defers the data-phase timing. Rejected: bring up the real thing once.
- **Stream buffer for RX transport:** stream buffers are for byte streams (single
  producer/consumer); CAN frames are discrete messages. A **queue** is the right fit. A future
  UART path will add a stream buffer alongside.
- **RX buffers instead of an RX FIFO:** dedicated RX buffers suit fixed known IDs, but a FIFO
  tolerates bursts and unknown arrival order better for a gateway. FIFO 0 kept.

## To verify in the TRM / configurator / on silicon

- Nominal and data bit-timing register values and the achieved sample points.
- Message-RAM section offsets/sizes fit the part's CANFD message RAM.
- `RF0N` is the correct new-message source and the write-1-clear semantics on this IP.

## Review history

Design-reviewed (`docs/review/ADR-0010-0011-review.md`) before implementation, then corrected
against silicon during M2 bring-up. Findings actioned here: **4** — the message-RAM footprint
made an explicit sizing constraint (D2); **S2** (silicon) — the TRAVEO CM4 **NVIC mux**:
`intrSrc` packs a mux channel (`<< CY_SYSINT_INTRSRC_MUXIRQ_SHIFT`) with the system interrupt and
`NVIC_EnableIRQ` takes the **mux channel**, not the bare IRQn — the bare form gives "init OK, no
RX" (D3); **S3** (silicon) — RX FIFO 0 must have a non-zero element count or every accepted frame
is dropped (D2; see `docs/briefs/M2-bringup_log.md`).
